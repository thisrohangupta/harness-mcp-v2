## Directory Purposes

**src/ — Source Code:**
- Purpose: All TypeScript source code organized by architectural layer
- Contains: Config, client, auth, registry, tools, resources, prompts, utilities
- Key files: `index.ts` (entrypoint), `config.ts` (validation), `client/harness-client.ts` (HTTP abstraction)

**src/auth/ — Authentication:**
- Purpose: JWT validation, principal extraction, auth middleware
- Contains: `index.ts` (exports), `principal.ts` (types), `jwt.ts` (validator), `middleware.ts` (Express middleware)
- Key exports: `JwtValidator`, `createJwtAuthMiddleware`, `AuthContext`, `Principal`

**src/client/ — HTTP Client:**
- Purpose: Unified HTTP abstraction for all Harness API communication
- Contains: `harness-client.ts` (main client class), `types.ts` (request/response types), `dto/` (API response shapes)
- Key class: `HarnessClient` — handles auth, retries, rate limiting, error normalization

**src/registry/ — Resource Registry & Dispatch:**
- Purpose: Declarative resource definitions and polymorphic operation dispatch
- Contains: `index.ts` (Registry class), `types.ts` (type defs), `extractors.ts` (response normalizers), `toolsets/` (27 domain files)
- Key class: `Registry` — caches toolset definitions, dispatches tool operations to endpoints, filters by env var

**src/registry/toolsets/ — Domain Definitions (27 files):**
- Purpose: Declaratively specify each Harness domain's resources and operations
- Pattern: Each file exports a `ToolsetDefinition` with resources array
- Examples:
  - `pipelines.ts`: pipeline, execution, trigger, input_set, approval resources
  - `services.ts`: service resource with list/get/create/update/delete operations
  - `environments.ts`: environment resource
  - `connectors.ts`: connector resource + test_connector action
  - `ccm.ts`: cost management resources (perspective, rule, commitment, etc.)
  - `audit.ts`: audit log resources
  - `feature-flags.ts`: feature flag resources + toggle action
  - 20+ more specialized domains

**src/tools/ — MCP Tool Implementations:**
- Purpose: Generic MCP tool handlers that dispatch to registry
- Contains: 11 tool files + 1 diagnose subdir
- Tools:
  - `harness_list`: List resources with filtering, pagination, optional visualization
  - `harness_get`: Get single resource by ID
  - `harness_create`: Create new resource with body
  - `harness_update`: Update existing resource
  - `harness_delete`: Delete resource (with confirmation gate)
  - `harness_execute`: Execute resource (pipelines, approvals, etc.)
  - `harness_diagnose`: Detailed failure analysis (subdir: `diagnose/{pipeline.ts, connector.ts, delegate.ts, gitops-application.ts}`)
  - `harness_search`: Full-text search across resources
  - `harness_describe`: Metadata about a resource type (filters, body schema, operations)
  - `harness_status`: Status + recent history of a resource
  - `harness_ask`: AI-powered natural language queries

**src/resources/ — MCP Resource Definitions:**
- Purpose: Read-only data that agents can reference without calling tools
- Contains: `pipeline-yaml.ts`, `execution-summary.ts`, `harness-schema.ts`
- Examples:
  - Pipeline YAML resource: `pipeline://{orgId}/{projectId}/{pipelineId}` → YAML as text
  - Execution summary: `executions://{orgId}/{projectId}/recent` → last 10 executions as JSON
  - Schema resource: `harness-schema://types` → all available resource types + operations

**src/prompts/ — MCP Prompt Templates:**
- Purpose: Curated multi-turn prompts for common tasks
- Contains: 25+ prompt files organized by domain
- Domains:
  - **Core**: `debug-pipeline.ts`, `create-pipeline.ts`, `optimize-costs.ts`, `security-review.ts`, `onboard-service.ts`
  - **DevOps**: `dora-metrics.ts`, `setup-gitops.ts`, `chaos-resilience.ts`, `feature-flag-rollout.ts`, `migrate-to-template.ts`, `delegate-health.ts`, `developer-scorecard.ts`
  - **FinOps**: `cloud-cost-breakdown.ts`, `commitment-utilization.ts`, `cost-anomaly.ts`, `rightsizing.ts`
  - **DevSecOps**: `vulnerability-triage.ts`, `sbom-compliance.ts`, `supply-chain-audit.ts`, `exemption-review.ts`, `access-control-audit.ts`
  - **Harness Code**: `code-review.ts`, `pr-summary.ts`, `branch-cleanup.ts`
  - **Approvals**: `pending-approvals.ts`
  - **Workflows**: `build-deploy-app.ts`

**src/utils/ — Utilities:**
- Purpose: Cross-cutting concerns (logging, errors, formatting, parsing, rendering)
- Contains: `logger.ts`, `errors.ts`, `response-formatter.ts`, `url-parser.ts`, `type-guards.ts`, `deep-links.ts`, `rate-limiter.ts`, `cli.ts`, `compact.ts`, `svg/` (chart rendering)
- Key utilities:
  - `logger.ts`: Structured JSON logging to stderr only
  - `errors.ts`: Error normalization + MCP error mapping
  - `response-formatter.ts`: JSON/text/mixed result helpers
  - `svg/`: SVG chart generation for timeseries/pie/bar visualizations
  - `rate-limiter.ts`: Token bucket rate limiting
  - `url-parser.ts`: Harness UI URL parsing (auto-extract org/project/resource)

**tests/ — Test Files:**
- Purpose: Unit and integration tests
- Pattern: Co-located with source (e.g., `tests/tools/harness-list.test.ts` for `src/tools/harness-list.ts`)
- Contains: Tool tests, client tests, integration tests

**build/ — Compiled Output:**
- Purpose: Generated JavaScript (not committed)
- Pattern: Mirrors src/ structure as .js files
- Generated by: `pnpm build` (tsc)
- Entry point: `build/index.js` (executable via shebang)

**docs/ — Documentation:**
- Purpose: Supplementary docs beyond README
- Contains: API reference, development guide, deployment docs

**k8s/ — Kubernetes:**
- Purpose: Deployment manifests for Kubernetes
- Contains: YAML files for StatefulSet, Service, ConfigMap, Secret, Ingress

**scripts/ — Build Tools:**
- Purpose: Development and build automation
- Contains: TypeScript compilation helpers, Docker build scripts

## Key File Locations

**Entry Points:**
- `src/index.ts`: Main server entrypoint — parses CLI args, loads config, starts stdio or HTTP transport

**Configuration:**
- `src/config.ts`: Zod schema for environment validation (HARNESS_API_KEY, JWT_SECRET, base URL, etc.)
- `src/auth/jwt.ts`: JWT validator implementation (signature verification, claim extraction)
- `src/auth/middleware.ts`: Express middleware for JWT authentication

**Core Logic:**
- `src/client/harness-client.ts`: HTTP client — auth header injection, retries, rate limiting
- `src/registry/index.ts`: Registry class — toolset loading, resource lookup, dispatch logic
- `src/registry/extractors.ts`: Response normalizers for different Harness API formats

**Tool Implementations:**
- `src/tools/harness-list.ts`: Generic list tool (dispatch + filtering + visualization)
- `src/tools/harness-get.ts`: Generic get tool
- `src/tools/harness-create.ts`: Generic create tool
- `src/tools/harness-describe.ts`: Metadata tool (resource types, filters, operations)
- `src/tools/harness-diagnose.ts`: Failure analysis (pipelines, connectors, delegates, GitOps)

**Testing:**
- `tests/tools/`: Unit tests for each tool
- `tests/client/`: Client tests
- `tests/integration/`: End-to-end tests with mock Harness API

## Naming Conventions

**Files:**
- Tool files: `harness-{action}.ts` (e.g., `harness-list.ts`, `harness-execute.ts`)
- Toolset files: `{domain}.ts` (e.g., `pipelines.ts`, `connectors.ts`)
- Resource files: `{resource-name}-resource.ts` (e.g., `pipeline-yaml.ts`)
- Prompt files: `{action}-{domain}.ts` (e.g., `debug-pipeline.ts`, `optimize-costs.ts`)
- Test files: `{subject}.test.ts` (e.g., `harness-list.test.ts`)

**Functions/Classes:**
- Camel case for functions/methods: `buildUrl()`, `handleRequest()`, `registerTool()`
- Pascal case for classes/types: `HarnessClient`, `Registry`, `ToolsetDefinition`
- Const exports for instances: `pipelinesToolset`, `servicesToolset`

**Constants:**
- Upper snake case: `BASE_BACKOFF_MS`, `SESSION_TTL_MS`, `RETRYABLE_STATUS_CODES`

**Directories:**
- Lower kebab case: `src/registry/toolsets/`, `src/utils/svg/`
- Singular for generic: `src/client/`, `src/auth/`, `src/tools/`
- Plural for collections: `src/tools/`, `src/prompts/`, `src/resources/`

## Where to Add New Code

**New Tool:**
1. Create `src/tools/harness-{action}.ts`
2. Export `register{Action}Tool(server, registry, client, config?)` function
3. Define Zod input schema with `.describe()` for each param
4. Implement tool handler with try/catch wrapping, error mapping
5. Add import + call to `src/tools/index.ts` (`registerAllTools` function)
6. Add test file: `tests/tools/harness-{action}.test.ts`

**New Toolset/Domain:**
1. Create `src/registry/toolsets/{domain}.ts`
2. Define `{domain}Toolset: ToolsetDefinition` with resources array
3. For each resource, specify:
   - `resourceType` (must be unique)
   - `operations` map with list/get/create/update/delete specs (as needed)
   - `listFilterFields` (for list operation)
   - `identifierFields` (e.g., ["pipeline_id"])
   - `deepLinkTemplate` to Harness UI
4. Add import + export to `src/registry/index.ts` (ALL_TOOLSETS array)
5. Update HARNESS_TOOLSETS config example in `.env.example`

**New Resource:**
1. Create `src/resources/{name}.ts`
2. Export `register{Name}Resource(server, registry, client, config)` function
3. Call `server.resource(uri, description, handler)` with URI template and async handler
4. Handler returns `{ contents: [{ uri, mimeType, text }] }`
5. Add import + call to `src/resources/index.ts` (`registerAllResources` function)

**New Prompt:**
1. Create `src/prompts/{action}-{domain}.ts`
2. Export `register{Action}Prompt(server)` function
3. Call `server.prompt(name, description, args, handler)`
4. Handler returns `{ messages: [{ role: "user", content: { type: "text", text: "..." } }] }`
5. Add import + call to `src/prompts/index.ts` (`registerAllPrompts` function)

**New Utility:**
1. Create `src/utils/{name}.ts`
2. Export typed functions/classes (prefer exports over defaults)
3. Import into layers that need it
4. Add unit tests if complex logic

## Special Directories

**src/registry/toolsets/**:
- Purpose: Declarative resource definitions (27 files, one per domain)
- Generated: No (hand-written)
- Committed: Yes
- Pattern: Each exports a `ToolsetDefinition` — pure data structure (no I/O, no HarnessClient calls)

**build/**:
- Purpose: Compiled JavaScript output
- Generated: Yes (by `pnpm build` running `tsc`)
- Committed: No (gitignored)
- How to regenerate: Run `pnpm build` or `pnpm dev` for watch mode

**src/utils/svg/**:
- Purpose: SVG chart rendering (timeseries, pie, bar)
- Generated: No (hand-written)
- Committed: Yes
- Example: `src/utils/svg/list-visuals.ts` generates inline PNG for execution stats

**tests/**:
- Purpose: Test files
- Generated: No (hand-written)
- Committed: Yes
- Pattern: Co-located mirrors src/ structure
- Run: `pnpm test` (Vitest)
